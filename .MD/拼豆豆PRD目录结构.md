## 4. 功能需求详细说明

### 4.1 图片导入模块

```
4.1.1 导入方式（相册选取 / 拍照）
4.1.2 支持的图片格式与尺寸限制
4.1.3 图片预处理（裁剪 / 主体识别 / 方向校正）
4.1.4 导入失败的异常处理
```

### 4.2 转图引擎（核心模块）

```
4.2.1 功能概述与核心目标
4.2.2 像素化算法需求
    - 输入：原始图片 + 目标尺寸
    - 输出：像素化网格图
    - 质量标准：轮廓清晰度 / 细节保留度

4.2.3 减色算法需求
    - 输入：像素化图 + 目标色数 + 指定色板
    - 输出：减色后的图稿（每像素对应一个色板色号）
    - 质量标准：感知色差 / 杂色控制 / 主体识别

4.2.4 杂色清理逻辑
    - 孤立点检测与自动合并规则
    - 用户可选的"优化"力度

4.2.5 三方案自动生成
    - 快速版定义（尺寸 / 色数 / 适用场景）
    - 标准版定义
    - 精细版定义
    - 各方案信息展示项：预览图 / 尺寸 / 色数 / 豆数 / 预计耗时 / 所需底板数

4.2.6 色板系统
    - Phase 1 支持的品牌色板列表
    - 色板数据结构（色号 / RGB / 颜色名称 / 品牌）
    - 默认色板逻辑 / 用户切换色板
    - 色板扩展机制

4.2.7 处理性能要求
    - 端侧处理 vs 云端处理决策
    - 目标耗时：< 3 秒
    - 设备兼容性要求

4.2.8 质量验收标准
    - 颜色准确度基线
    - 杂色率上限
    - 人工盲评通过率要求
```

### 4.3 方案预览与选择

```
4.3.1 预览界面布局
4.3.2 方案切换交互（滑动对比）
4.3.3 各方案信息展示规格
4.3.4 用户手动调参入口（二级操作）
    - 可调参数：尺寸 / 色数
    - 调参后实时预览
4.3.5 确认选择 → 进入下一步
```

### 4.4 图稿编辑器

```
4.4.1 编辑器界面布局
4.4.2 画布操作
    - 缩放 / 平移
    - 网格显示开关
    - 色号标注显示开关

4.4.3 编辑工具清单
    - 点选改色（点击单格 → 调出色板 → 选色替换）
    - 区域填色（长按拖选 → 填充）
    - 橡皮擦（清除为透明/底色）
    - 颜色替换（全局：A色 → B色）
    - 撤销 / 重做

4.4.4 一键优化按钮
    - 移除孤立杂色点
    - 合并相近色
    - 强化轮廓对比度
    - 可重复点击，渐进优化

4.4.5 编辑状态保存逻辑（自动保存 / 手动保存）
4.4.6 编辑器性能要求（大图流畅度）
```

### 4.5 材料清单

```
4.5.1 清单数据项
    - 颜色名称 + 色号 + 品牌
    - 各颜色所需颗数
    - 总豆数
    - 所需底板数量与规格
    - 成品尺寸（厘米）

4.5.2 排序方式（按颜色 / 按用量）
4.5.3 清单与图稿实时联动（编辑后自动更新）
4.5.4 清单独立导出（图片 / 文本）
```

### 4.6 大图分板

```
4.6.1 触发条件（作品尺寸超过单块底板）
4.6.2 自动拆板逻辑

4.6.3 分板视图 / 整体视图切换
4.6.4 各板独立导出 + 拼接顺序标注
```

### 4.7 图纸导出

```
4.7.1 导出格式
    - 高清图片（PNG）
    - 可打印 PDF

4.7.2 图片导出规格
    - 彩色版（网格 + 颜色填充 + 色号标注）
    - 黑白色号版（网格 + 色号文字，省墨打印）
    - 水印逻辑（免费版带水印 / 付费版无水印）

4.7.3 PDF 打印优化
    - A4 自动排版
    - 大图分页 + 页码 + 拼接边标记
    - 每页包含：图稿区 + 颜色图例 + 作品信息

4.7.4 导出物模板与视觉规格（附示意图）

4.7.5 分享能力
    - 系统分享（微信 / 保存相册等）
```

### 4.8 作品库

## 目录结构设计说明

| 设计决策 | 理由 |
|----------|------|
| **转图引擎单独拆为最大章节** | 这是产品胜负手，需要需求、质量标准、验收标准全部写清楚 |
| **用户流程独立成章** | 避免流程逻辑散落在各功能模块中，团队对端到端体验缺乏整体认知 |
| **付费设计独立成章** | 付费触发节点涉及多个功能模块的联动，集中管理更清晰 |
| **埋点与指标独立成章** | 确保每个功能上线前就定义好"怎么衡量好不好"，不是上线后补 |
| **Phase 2/3 放附录而非正文** | MVP 文档必须聚焦，未来规划仅作参考，不干扰当前开发判断 |
| **不设"社区"和"库存"章节** | Phase 1 明确不做，不在 PRD 中占位，避免团队产生"迟早要做"的暗示 |
